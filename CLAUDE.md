# TypingShooter — プロジェクトルール

迫りくる敵にロックオンし、頭上のお題をタイピングして撃ちまくる単一HTMLタイピングシューティング。YouTube Playable 向け。

## 進捗記録ルール（最重要）

**作業をしたら、必ず `Progress.md` に進捗を記録すること。**

- コード追加・修正、ファイル生成、設定変更、公開作業など、何らかの作業を行うたびに `Progress.md` へ追記する。
- 各エントリには「日付 / 作業内容 / 変更ファイル / 次にやること」を含める。
- 新しい情報が出てきたら古い記述を更新し、最新状態を保つ。
- セッションを開始したら、まず `Progress.md` を読んで前回までの状況を把握してから作業を始める。

## プロジェクト概要

- **ゲームジャンル**: タイピングシューティング（ロックオン式）
- **配信先**: YouTube Playable（仕様書では Google Playable / 汎用 Playable を想定）
- **成果物**: `TypingShooter.html`（単一HTML完結・ビルド不要・外部素材/外部通信なし）
- **技術**: 素の JavaScript / Canvas 2D（描画）/ Web Audio API（効果音は全て合成）/ DOMオーバーレイ（メニューUI）
- **詳細仕様**: `TypingShooter_仕様書.md` を参照（モード・操作・ローマ字エンジン・難易度式・データ・アーキテクチャ）。**作業前に必ず読むこと。**

## 変更注意箇所（回帰しやすい）

- **ローマ字エンジン（`RomaTyper`）**: 複数表記（shi/si・cha/tya）・促音・撥音の受理ルール。変更したら全語完走テスト（`test5.js`）を必ず回す。
- **タッチ端末判定**: `matchMedia('(pointer: coarse)')` で主ポインタが指の端末のみタッチ扱い。`maxTouchPoints>0` に戻すとタッチスクリーン付きPCでキー入力が死ぬ。
- **デバッグフック `window.TF`**: 自動テストが使用。削除・改名しないこと。
- **お題データ追加時**: 日本語はひらがなのみ（長音「ー」・カタカナ・記号不可）。英語は小文字+スペースのみ。追加後は `test5.js` で全語完走を確認。

## ワークフロー

1. 仕様の確認（`TypingShooter_仕様書.md`）と `Progress.md` の読み込み
2. `TypingShooter.html` の実装・修正
3. 回帰テスト（最低限 `test.js` と `test5.js`。`npm test` で実行）
4. GitHub リポジトリへコミット・プッシュ
5. 各ステップ完了ごとに `Progress.md` へ記録

## YouTube Playable 向けの注意点

- 単一 HTML／軽量構成を基本とし、外部依存・外部通信をしない（審査要件）。
- モバイル縦横・タッチ操作を前提にレスポンシブ対応する（全アスペクト比対応は審査要件）。論理 960×540 を CSS でフィット。
- ロード時間と初回描画を軽くする（アセットは全てコード生成済み・画像素材なし）。
- 進行の永続化は現状メモリ保持のみ（`localStorage` 不使用）。配信先 SDK のストレージ要件に合わせて保存処理を追加する余地あり。
- 公式 Playable SDK（`ytgame` / `mraid.js` 等）を使う場合はゲームコードより先に読み込み、承認前はスタンドアロン用フォールバックで動かす（未実装・引き継ぎTODO）。
